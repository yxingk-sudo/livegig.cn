## 权限体系结构

### 后台权限层级

```
超级管理员 (super_admin)
    │
    ├─ 拥有所有后台权限
    ├─ 可以访问所有公司和项目
    ├─ 可以配置所有角色的权限
    └─ 可以管理系统配置
    
管理员 (admin)
    │
    ├─ 基于公司维度的权限
    ├─ 可以访问所属公司的所有项目
    ├─ 可以管理公司内的用户和项目
    └─ 权限由超级管理员配置
    
项目管理员 (project_admin)
    │
    ├─ 限定于指定项目
    ├─ 只能访问被授权的项目
    ├─ 可以管理项目内的数据
    └─ 权限由超级管理员或管理员配置
```

### 前台权限层级

```
前台管理员 (user_admin)
    │
    ├─ 可以配置前台用户权限
    ├─ 可以管理项目内的用户
    ├─ 拥有项目内的所有功能权限
    └─ 权限由后台管理员配置
    
前台用户 (user)
    │
    ├─ 基于分配的权限集
    ├─ 只能访问授权的页面和功能
    ├─ 界面动态显示可用功能
    └─ 权限由前台管理员或后台管理员配置
```

## 数据库表关系

```
roles (角色表)
    │
    ├─> role_permissions (角色权限关联)
    │       │
    │       └─> permissions (权限资源表)
    │
    ├─> admin_users (后台用户)
    │       │
    │       └─> admin_user_projects (用户项目关联)
    │               │
    │               └─> projects (项目表)
    │
    └─> user_roles (前台用户角色)
            │
            └─> project_users (前台用户)

permissions (权限资源表)
    │
    ├─ parent_id (自关联，构建树形结构)
    │
    └─> user_custom_permissions (用户自定义权限)

permission_logs (权限操作日志)
    │
    └─ 记录所有权限相关操作
```

## 权限验证流程

### 页面访问流程

```
用户访问页面
    │
    ▼
┌────────────────────┐
│ 检查登录状态       │
└────────────────────┘
    │
    ▼
┌────────────────────┐
│ 获取用户角色       │
└────────────────────┘
    │
    ▼
┌────────────────────┐
│ 检查页面权限       │
│ (checkPagePermission)│
└────────────────────┘
    │
    ├─ 有权限 ──────> 显示页面
    │
    └─ 无权限 ──────> 显示拒绝访问页面/重定向登录
```

### 功能验证流程

```
用户点击功能按钮
    │
    ▼
┌────────────────────┐
│ 前端权限检查       │
│ (hasPermission)    │
└────────────────────┘
    │
    ├─ 无权限 ──────> 隐藏/禁用按钮
    │
    └─ 有权限
        │
        ▼
    ┌────────────────────┐
    │ 调用后端API        │
    └────────────────────┘
        │
        ▼
    ┌────────────────────┐
    │ 后端权限验证       │
    │ (checkApiPermission)│
    └────────────────────┘
        │
        ├─ 无权限 ──────> 返回错误响应
        │
        └─ 有权限
            │
            ▼
        ┌────────────────────┐
        │ 检查数据访问权限   │
        │ (checkDataAccess)  │
        └────────────────────┘
            │
            ├─ 无权限 ──────> 返回错误响应
            │
            └─ 有权限 ──────> 执行操作并返回结果
```

### 权限判断优先级

```
1. 自定义权限 (user_custom_permissions)
   │
   ├─ grant (授予) ────> 允许访问
   └─ deny (拒绝)  ────> 拒绝访问
   
2. 角色权限 (role_permissions)
   │
   ├─ 有权限记录 ────> 允许访问
   └─ 无权限记录 ────> 拒绝访问
```

## 权限控制三个维度

### 1. 页面级权限

```
permission_type = 'page'

示例：
- backend:project:list      (项目列表页面)
- frontend:personnel:list   (人员列表页面)
- backend:system:permission (权限管理页面)
```

### 2. 功能级权限

```
permission_type = 'function'

示例：
- backend:project:add       (添加项目功能)
- backend:project:edit      (编辑项目功能)
- backend:project:delete    (删除项目功能)
- frontend:meal:batch       (批量报餐功能)
```

### 3. 数据级权限

```
permission_type = 'data'

通过用户角色和所属范围控制：
- 超级管理员：所有数据
- 管理员：所属公司的数据
- 项目管理员：授权项目的数据
- 前台用户：项目内个人或团队数据
```

## 安全策略

### 前后端双重验证

```
前端控制 (用户体验)
    │
    ├─ 隐藏无权限的菜单和按钮
    ├─ 禁用无权限的操作
    └─ 提供友好的权限提示
    
后端验证 (安全保障)
    │
    ├─ 页面访问权限验证
    ├─ API调用权限验证
    ├─ 数据访问权限验证
    └─ 防止直接URL访问和API调用
```

### 不可见即无权限原则

```
1. 界面层
   └─ 无权限的元素完全隐藏或禁用

2. API层
   └─ 严格验证每个API调用的权限

3. 数据层
   └─ 基于角色和范围过滤查询结果
```

## 扩展性设计

### 添加新权限

```sql
-- 1. 添加新权限
INSERT INTO permissions (...) VALUES (...);

-- 2. 为角色分配权限
INSERT INTO role_permissions (role_id, permission_id) VALUES (...);

-- 3. 在代码中使用
requireAdminPermission('new:permission:key');
```

### 添加新角色

```sql
-- 1. 创建新角色
INSERT INTO roles (role_name, role_key, role_type, description) 
VALUES ('新角色', 'new_role', 'backend', '角色描述');

-- 2. 分配权限
INSERT INTO role_permissions (role_id, permission_id) 
SELECT LAST_INSERT_ID(), id FROM permissions WHERE ...;

-- 3. 创建该角色的用户
INSERT INTO admin_users (..., role_id, ...) 
VALUES (..., LAST_INSERT_ID(), ...);
```

### 自定义权限规则

```php
// 为特定用户设置特殊权限
$permissionManager->setCustomPermission(
    $userId,
    $userType,
    $permissionId,
    'grant', // 或 'deny'
    $operatorId,
    $operatorType
);
```

## 性能优化建议

### 1. 权限缓存

```php
// Session缓存
$_SESSION['user_permissions'] = $permissions;

// Redis缓存
$redis->setex("user_perm:{$userId}", 3600, json_encode($permissions));
```

### 2. 减少数据库查询

```php
// 一次性获取所有权限
$allPermissions = $permissionManager->getUserPermissions($userId, $userType, $projectId);

// 在内存中检查
foreach ($allPermissions as $perm) {
    if ($perm['permission_key'] === $targetKey) {
        return true;
    }
}
```

### 3. 使用索引

```sql
-- 已在数据库设计中添加必要索引
CREATE INDEX idx_permission_key ON permissions(permission_key);
CREATE INDEX idx_role_id ON admin_users(role_id);
CREATE INDEX idx_user_type ON user_custom_permissions(user_id, user_type);
```

## 总结

本权限管理系统具有以下特点：

✅ **完整性**：覆盖前后台、页面、功能、数据三个维度
✅ **安全性**：前后端双重验证，防止越权访问
✅ **灵活性**：支持角色权限、自定义权限、权限继承
✅ **可扩展性**：模块化设计，易于添加新权限和角色
✅ **易用性**：可视化配置界面，完整的文档和示例
✅ **可维护性**：清晰的代码结构，完善的日志审计

该系统满足企业级应用的权限管理需求，可根据实际业务场景灵活配置和扩展。
